## #8 ROOT CAUSE (the data-integrity bug)

**React Hook Form stores EDITED numeric `<input>` values as STRINGS** (e.g. `"25"`); only the seeded `DEFAULT_CONFIG` values are real `number`s. `tallyCatalog` guarded with `Number.isFinite(value)` — and `Number.isFinite("25")` is **`false`** — so **every edited quantity/weight silently collapsed to `0`** in the displayed badge + footer. Meanwhile the submit path (the zod schema / `buildPackRequest`) coerces the same string to `25`, so the **built `PackRequest` packed 25 while the UI showed 0**. That single coercion mismatch is both the displayed-vs-submitted divergence (#8) and the "stale counter" symptom (#5) — the counter wasn't stale, it was wrong the instant a field was edited.

Investigated and **ruled out**: (a) `useFieldArray` `fields`-snapshot vs live drift — `fields.length` tracked `boxTypes.length` exactly across non-tail add/remove; (b) `useWatch('boxTypes')` nested-subscription staleness — a clean `useWatch` probe re-fired correctly on nested edits; (c) the autosave restore/subscribe path — it never resets the form, and `getValues()` returned correct live values throughout.

**Fix:** `tallyCatalog` now runs the SAME `Number(v)` coercion as the request builder (`toFiniteNumber`), so the rendered rows, the badge, the footer, and the built `PackRequest` are provably one source of truth.

**How the regression pins it:** `BoxCatalogCard.test.tsx` mounts `BoxCatalogCard` + `FooterBar` in one form, drives an add/edit(non-tail)/remove/add sequence, then asserts `displayedRows === tally.types === builtTypes` AND `tally.units === request.boxes.length` AND that the badge/footer text equals the tally. On the pre-fix tally the assertion fails (`20` vs `33` — edited string rows count 0 while the builder expands the real count); after the fix it passes (33 units / 3 types). Verified by temporarily reverting `config-tally.ts` to the `Number.isFinite` guard (test failed) and restoring (test passed).

## Camera fix (#11)

The tilt-then-snap came from lerping position + target linearly while the look-direction (orientation/up) changes nonlinearly between presets (TOP looks straight down; ISO is three-quarter), so the framing tilted gradually then the rotation resolved abruptly at the end. `camera-presets.ts` gains pure, three-free, jsdom-testable helpers — `lookQuaternion(position, target, up)` (orthonormal-basis → quaternion via Shepperd's method), `slerpQuat(a, b, t)`, `quatAngle`, and `presetQuaternion(bbox, which)`. `CameraPresets.tsx` captures from/to look-quaternions at transition start and **slerps `camera.quaternion` alongside the eased position/target lerp**, applied after `controls.update()` so it overrides OrbitControls' linearly-derived look-at FOR THE DURATION; on the settled frame (`k>=1`) the slerp is skipped so `controls.update()` owns the final orientation and OrbitControls' spherical state stays consistent for subsequent drags. Preserved: `bboxRef`/`measureNonce` camera-preservation-on-switch (D-02), the `__cameraState` hook, `presetFromBbox` framing distances, and the OrbitControls clamps. New tests pin ISO≠TOP orientation and an even t=0.5 arc bisection (uniform sweep).
