# Phase 6 вҖ” UI Design Contract

> Visual and interaction contract for the **Result screen** (lazy three-only `/result` chunk). Generated by gsd-ui-researcher from `design/result.html` (the visual north star) + 06-CONTEXT.md (authoritative user decisions). Verified by gsd-ui-checker.

> **Source-of-truth note:** This phase **feeds real data into the already-built Phase-2 scene + chrome and adds the right rail** вҖ” it does NOT rebuild the viewer (C-01). The light rail tokens (`--bg`, surfaces, borders, text, accent group, `--radius-*`, `--shadow`) and the dark 3D-overlay group (`--d-bg/-border/-text/-text-2`) are already ported into `src/styles.css` `@theme` (Phases 2/4). This phase ports only the **two missing result-rail tokens** from the mockup `:root` (`--color-pos: #16a34a`, `--color-warn: #d97706`) вҖ” see Token Porting Checklist. The dark overlay group and all light tokens stay untouched. `/result` is the LAZY three-only chunk (C-04); the rail is plain DOM but lives inside the lazy subtree вҖ” fine.

> **Honest-over-pretty mandate (C-05 / D-04 / D-07):** No fabricated data. **No** "Solved in 1.84s" pill (the `DoneResponse` carries no timing field). **No** Export button (Phase 7). **No** amber low-fill "warn" treatment on pallet rows вҖ” fill% is shown neutrally; the solver decides placement and we never imply a quality judgment it didn't make.

## Spacing Scale

The mockup uses a near-8pt rhythm with a few odd values (rail-block padding `24px 26px`, summary gap `22px 18px`, overlay offsets `26px/30px`). For the port, **snap everything to the standard scale** while preserving the mockup's visual proportions. The declared scale uses ONLY standard-set values:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Swatch/dot gaps, stat-keyвҶ”value gap, mini fill-bar top margin |
| sm | 8px | Pallet-row internal gap, legend item gap, placement-card top-row gap, switcher row gap |
| md | 16px | Block-label bottom margin, placement-card list gap, summary col gap (mockup 18px вҶ’ 16), pallet-row padding |
| lg | 24px | Rail-block padding (mockup `24px 26px` вҶ’ 24), summary row gap (mockup 22px вҶ’ 24), card-to-content rhythm |
| xl | 32px | Reserved for major intra-rail breaks |
| 2xl | 48px | Reserved for empty/centered-state vertical padding |
| 3xl | 64px | Reserved for page-level breaks |

**Snap rule:** where the mockup uses 26px / 22px / 18px / 30px gaps, planner snaps to the nearest standard token (24 / 24 / 16 / 32). Visual parity preserved; no new spacing tokens introduced.

## Typography

Ported from the mockup. **Exactly 4 sizes + exactly 2 weights** form the contract вҖ” consistent with the Phase 4 contract so the two screens share one type system. The mockup's micro-sizes (10.5 / 11.5 / 12.5px) and intermediate weights (450/500) collapse into these roles. Planner uses the role, never a decimal size or intermediate weight.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (summary stat value, e.g. `78.4`) | 22px | 600 (semibold) | 1.1 |
| Heading (block scope `Result`, pallet name, placement id, viewer title) | 14px | 600 (semibold) | 1.3 |
| Body / row text (pallet name, type name, unpacked reason) | 14px | 400 (regular) | 1.5 |
| Mono value (dims, position, weight, fill%, support%, ids, legend, overlay sub-line) | 13px | 400 (regular) | 1.4 |
| Label / meta / block-label / cell-label / overlay tag / hint | 12px | 600 (block-label) / 400 (meta, hint, tag) | 1.4 |

**Declared font sizes (exactly 4):** **22** (stat value / display), **14** (heading + body), **13** (mono value), **12** (label / meta / block-label / overlay tag / cell-label / hint). No other sizes permitted. (Mockup `24px` stat value вҶ’ snaps to the existing `22` display size for cross-screen parity; mockup `10.5/11/11.5` micro-sizes вҶ’ `12`.)

**Declared font weights (exactly 2):**
- **400 (regular)** вҖ” body/row text, all mono values, hints, overlay sub-line, legend, cell labels, type names.
- **600 (semibold)** вҖ” headings (viewer title, pallet name, placement id), stat values, block labels (mono uppercase), the active step, and button text (`Edit configuration`).

No intermediate weight (450/500). Mockup 450/500 вҶ’ **600** if heading/label/button/value, otherwise **400**.

**Mono usage (JetBrains Mono вҖ” 400/600 only):** every numeric value (stat values 22px/600; dims/position/weight/fill%/support% at 13px/400), ids, the legend rows, the overlay dims tag + computed sub-line, and the block labels (uppercase 12px). `font-variant-numeric: tabular-nums` on all numeric mono so digits align in lists/stats.

Base body: **14px / line-height 1.5 / weight 400**, `letter-spacing: -0.006em`. Headings: `letter-spacing: -0.01em` to `-0.02em`. Dark-overlay text uses the `--color-d-text` / `--color-d-text-2` group (already tokenised).

## Color

The Result screen is **dual-surface**: a dark 3D viewer (left) and a light rail (right). Both token groups are already in `@theme`. The 60/30/10 split is interpreted per-surface.

### Light rail (the rail + topbar вҖ” the screen's measurable chrome)

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` (`--surface`) | Rail background, topbar (rgba white .85 + blur), card surfaces. The rail is the persistent reading surface. |
| Secondary (30%) | `#f7f8fa` (`--bg`) + `#fcfcfd` (`--surface-2`) | Page base behind cards, pallet-row hover (surface-2), index-chip neutral fill (`#eceef1`), stat fill-bar track (`#edeef1`). |
| Accent (10%) | `#4f46e5` (`--accent`) | See reserved-for list below. |
| Destructive | `#dc2626` (`--danger`) | Not used as a fill this phase вҖ” no destructive action on the Result screen (see Copywriting). Reserved/available; not applied. |

### Dark viewer overlay (already tokenised вҖ” do not retint)

| Role | Value | Usage |
|------|-------|-------|
| Dominant | `#0c0f17` (`--d-bg`) | Scene background / fog (radial-gradient base in the mockup). |
| Overlay border | `#222838` (`--d-border`) | Overlay tag/button borders. |
| Overlay text | `#e6e8ee` (`--d-text`) primary / `#838b9e` (`--d-text-2`) secondary | Viewer title (primary), dims tag / sub-line / legend / hints / axis buttons (secondary). |

**Accent (`#4f46e5`) reserved EXCLUSIVELY for:**
1. The **active step** indicator in the topbar step-nav (`Result` filled circle + white number).
2. The **selected pallet row** in the switcher (`--accent` border + `--accent-weak` `#ecebfd` bg + inset accent ring; selected index chip flips to `--accent` fill / white text).
3. The **hovered/highlighted placement card** (`--accent` border + `--accent-weak` bg) вҖ” the visual half of the rowвҶ’mesh hover link (D-11).
4. The **summary Utilisation fill bar** and the **selected pallet's mini fill bar** (accent fill on a neutral track).
5. The **rotation tag** on placement cards (`--accent-text` `#4338ca` on `--accent-weak`).
6. The **brand glyph** gradient (`linear-gradient(150deg,#6d63f5,#4f46e5)`) and the active axis button (dark-overlay: `rgba(99,90,245,.32)` fill вҖ” a scene-overlay accent, already in chrome).

**Semantic positive (`--pos` `#16a34a`, NEW token this phase):** reserved for the **"All items packed вң“"** affordance when `unpacked_items.length === 0` (D-06) вҖ” a success checkmark/text, NOT a fill. Never used for accent purposes.

**`--warn` (`#d97706`, ported but DELIBERATELY UNUSED on pallet rows):** Per **D-04 the amber low-fill warn treatment is DROPPED** вҖ” pallet fill% renders **neutrally** regardless of value. The token is ported for token-parity with the mockup `:root` (and possible future use), but **no pallet row, fill bar, or stat may render amber based on a client-side fill threshold this phase.** The optional large-job advisory styling (if surfaced) is the only sanctioned amber, and that lives on the Configure screen, not here.

**Never** apply accent to: body/row text, every row border, hint text, the per-type swatches, or non-selected pallet rows. The **per-type swatch / box-tint / legend colours are NOT design tokens** вҖ” they come from `colorForType(typeId)` (`src/lib/palette.ts`), the single shared map for legend + box mesh tint + placement-card swatch + pallet-row swatch. Do NOT hard-code the mockup seed colours (`#6d63f5/#0ea5a3/#e0892b`).

**Support-ratio heatmap (DIAG-02 / D-10) вҖ” alternate box colouring, OFF by default:** an opt-in toggle recolours box meshes by a support-ratio scale (well-supported вҶ’ low-support). This is a **scene-material colour scale**, computed by a pure `src/lib/` bucketing function, NOT a UI token and NOT the `--warn`/`--danger`/`--pos` tokens. Default box colouring stays **by-type** (the legend). When the heatmap is ON, the legend swaps to a support scale key. The scale must be perceptually ordered and colour-blind-considerate (avoid pure red/green as the only signal вҖ” pair with the numeric `support N%` on the card, which is always shown).

## Accessibility Conventions

| Control | Convention |
|---------|------------|
| Pallet switcher rows | Each row is a `<button>` (or `role="button"` + keyboard). The switcher is a single-select group: exactly one selected (`aria-pressed`/`aria-current`), arrow/Tab navigable, Enter/Space selects. Selected state must NOT rely on colour alone вҖ” the accent border + inset ring + filled index chip provide non-colour cues. |
| Placement row вҶ’ mesh hover (D-11) | Hover is a **progressive enhancement**, not the only path to the data: every placement datum (id, type, dims, position, weight, support%) is readable in the card text without hovering. The emissive mesh highlight is a visual aid; no information is hover-only. Keyboard focus on a card SHOULD mirror the hover highlight (focus = hover parity) where feasible. |
| CoG / Support-heatmap toggles | Role `switch` (or pressed button) with accessible name from the visible label (`Centre of gravity`, `Support heatmap`); reflect on/off via `aria-checked`/`aria-pressed`. |
| Axis buttons (ISO/TOP/FRONT) | Existing controls; the active preset uses `aria-pressed="true"` and a non-colour-only "on" cue. |
| Icon-only "Edit configuration" affordance | Has a visible text label, so no extra `aria-label` needed; the inline arrow icon is `aria-hidden="true"`. |
| Support-ratio signal | Never colour-only вҖ” the numeric `support N%` on every placement card is the primary signal; the heatmap is an optional visual overlay paired with the legend scale. |
| 3D canvas | The canvas is a visual exploration aid; the rail (placement list, summary, unpacked) is the **accessible textual equivalent** of the scene's data. This is the reason the rail must NOT be hidden on mobile (D-08) вҖ” it is the non-WebGL data path. |
