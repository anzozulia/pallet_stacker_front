<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| router navigation state → LoadingPage | The `{ request, idToType }` handoff crosses via react-router state; a direct deep-link to /loading with no state is a missing-input case (degrade gracefully, not crash). |
| poll status field → rendered sub-line | The API `status` string drives UI text; it is a closed validated union (zod, Wave 1) so it cannot inject arbitrary text. |
| code-split chunk boundary | /loading must not pull three into the eager chunk. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-07 | DoS (availability) | direct deep-link to `/loading` with no nav state | mitigate | LoadingPage reads `useLocation().state` defensively; with no `request` it has nothing to submit — degrade to a redirect to `/` (or a neutral empty state), never crash on `undefined`. (The destructive failure-card paths are Plan 04; this is just the no-input guard.) |
| T-5-08 | Tampering | code-split leak (three in the eager chunk) | mitigate | `/loading` is wired as an EAGER static import with NO three/viewer imports; the Wave-4 build gate (`scripts/check-code-split.mjs`) asserts three stays only in the lazy `/result` chunk (C-06). |
| T-5-09 | Information disclosure | status sub-line text | accept | Driven by a validated closed status union mapped to fixed strings; rendered as React text (auto-escaped) — no untrusted interpolation, no `dangerouslySetInnerHTML`. |
</threat_model>
