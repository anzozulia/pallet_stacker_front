<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| react-query hooks → fetch client | The hooks delegate all IO to the Wave-1 client; the AbortSignal flows hooks → client → fetch. |
| poll loop → wall-clock time | A server that never returns a terminal status is an availability risk; the client safety cap bounds it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-04 | DoS (resource exhaustion) | the poll loop in `usePollJob` | mitigate | `refetchInterval` returns `false` on terminal status (self-stopping); react-query clears its own interval + aborts the in-flight queryFn on disable/unmount (SC-3) — no hand-rolled `setInterval` (C-01). |
| T-5-05 | DoS (availability) | a never-terminal / stuck-`queued` job | mitigate | `POLL_SAFETY_CAP_MS` (~120-140s named constant) bounds the poll so a server that never returns a terminal status surfaces an error (timeout/unreachable bucket) instead of spinning forever (Pitfall 1). `retry:false` so a failing poll is not silently re-fired (masks unreachable). |
| T-5-06 | Tampering | mutation/query error surface | accept→mitigate | Errors propagate unmodified to the consumer (no swallow); the Wave-3 LoadingPage classifies them. The hooks themselves never `dangerouslySetInnerHTML` or interpolate untrusted text. |
</threat_model>
